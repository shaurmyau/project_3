# Отчёт: Аномалии изоляции в SQL (MySQL)

## 1. Dirty Read (грязное чтение)

**Уровень изоляции:** `READ UNCOMMITTED`

**Суть аномалии:**  
Одна транзакция читает данные, которые были изменены другой транзакцией, но ещё не зафиксированы. Если вторая транзакция выполнит откат, первая останется с несуществующими данными.

**Как было воспроизведено:**  
- Сессия 1 обновила баланс с 1000 на 2000, не делая `COMMIT`.  
- Сессия 2 (с уровнем `READ UNCOMMITTED`) выполнила `SELECT` и увидела 2000.  
- Сессия 1 выполнила `ROLLBACK`.  
- Сессия 2 осталась с «грязным» значением 2000, которого никогда не было в БД.

**Как избежать:**  
Переключиться на уровень изоляции `READ COMMITTED` или выше. При этом любая транзакция будет видеть только уже зафиксированные изменения.  
Демонстрационная команда:  
```sql
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;
```

## 2. Non-repeatable Read (неповторяемое чтение)

**Уровень изоляции:** `READ COMMITTED`

**Суть аномалии:**  
В рамках одной транзакции два одинаковых запроса возвращают разные значения, потому что между ними другая транзакция изменила и зафиксировала те же данные.

**Как было воспроизведено:**  
- Сессия 1 (уровень `READ COMMITTED`) начала транзакцию и прочитала баланс Alice = 1000.  
- Сессия 2 изменила баланс на 1500 и зафиксировала (`COMMIT`).  
- Сессия 1 снова прочитала баланс – получила 1500.  
- Внутри одной транзакции сессии 1 появилось два разных результата.

**Как избежать:**  
Использовать уровень `REPEATABLE READ` или `SERIALIZABLE`. Тогда транзакция видит снимок данных на момент своего начала.  
Демонстрационная команда:  
```sql
SET SESSION TRANSACTION ISOLATION LEVEL REPEATABLE READ;
```

## 3. Phantom Read (фантомное чтение)

**Уровень изоляции:** `REPEATABLE READ`

**Суть аномалии:**  
При повторном выполнении запроса с условием (`WHERE`) появляются новые строки (или исчезают старые), добавленные (удалённые) другой транзакцией. В MySQL на `REPEATABLE READ` фантомы возможны.

**Как было воспроизведено:**  
- Сессия 1 (уровень `REPEATABLE READ`) начала транзакцию и посчитала количество записей с балансом > 500 – их было 1 (Alice).  
- Сессия 2 вставила новую строку (Charlie, balance = 800) и зафиксировала.  
- Сессия 1 повторила тот же `COUNT` – получила 2 (появился Charlie).  
- Внутри транзакции количество строк, удовлетворяющих условию, изменилось – фантом.

**Как избежать:**  
Перейти на уровень `SERIALIZABLE`. Он блокирует диапазоны строк, не позволяя другим транзакциям вставлять или удалять строки, которые могли бы повлиять на текущий запрос.  
Демонстрационная команда:  
```sql
SET SESSION TRANSACTION ISOLATION LEVEL SERIALIZABLE;
```

## 4. Lost Update (потерянное обновление)

**Уровень изоляции:** любой (обычно `READ COMMITTED` или `REPEATABLE READ`)

**Суть аномалии:**  
Две транзакции одновременно читают одно и то же значение, каждая изменяет его на основе прочитанного, и последняя запись «затирает» результат первой. Это классическая проблема неатомарного обновления.

**Как было воспроизведено:**  
- Таблица `counters` содержала value = 10.  
- Сессия 1 прочитала 10, решила увеличить на 5 (новое = 15).  
- Сессия 2 тоже прочитала 10 (до того, как сессия 1 зафиксировала изменение), решила увеличить на 3 (новое = 13).  
- Сессия 1 зафиксировала 15, затем сессия 2 зафиксировала 13.  
- Итоговое значение 13 – обновление сессии 1 потеряно. Ожидалось 10+5+3 = 18.

**Как избежать:**  
- **Атомарное обновление:** вместо чтения и последующего вычисления выполнить `UPDATE counters SET value = value + 5 WHERE id = 1`.  
- **Блокировка строки:** прочитать с `SELECT ... FOR UPDATE`, тогда вторая транзакция будет ждать, пока первая не завершится.  

Демонстрационные команды:  
```sql
-- Атомарный вариант
UPDATE counters SET value = value + 5 WHERE id = 1;

-- Вариант с явной блокировкой
SELECT value FROM counters WHERE id = 1 FOR UPDATE;
UPDATE counters SET value = value + 5 WHERE id = 1;
COMMIT;
```

---

## Общий вывод

- **Dirty read** – самая грубая аномалия, устраняется переходом на `READ COMMITTED`.  
- **Non-repeatable read** – устраняется `REPEATABLE READ` или выше.  
- **Phantom read** – требует `SERIALIZABLE` (в некоторых СУБД `REPEATABLE READ` тоже защищает, но в MySQL – нет).  
- **Lost update** – лечится не уровнем изоляции, а правильным стилем записи (атомарные операции или блокировка).

В реальных проектах обычно выбирают `REPEATABLE READ` (как в MySQL по умолчанию) или `READ COMMITTED` (как в PostgreSQL), а для критичных операций используют явные блокировки `FOR UPDATE`.

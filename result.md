# Сравнительный анализ производительности RabbitMQ и Redis (равные условия)

## Условия тестирования
- **Producer**: 4 потока для обоих брокеров, каждый поток отправляет целевую долю сообщений (`rate / threads`).
- **Consumer**: однопоточный, `prefetch=5000` для RabbitMQ, `blpop` для Redis.
- **Размеры сообщений**: 128 Б, 1024 Б (1 КБ), 10240 Б (10 КБ), 102400 Б (100 КБ).
- **Целевые интенсивности**: 1000, 5000, 10000 msg/с.
- **Длительность теста**: producer работал 10 с, consumer — ~15 с (время ожидания).

---

| Размер | Rate   | RabbitMQ | Redis   | Δ (Redis – RabbitMQ) |
|--------|--------|----------|---------|----------------------|
| 128 Б  | 1000   | 649      | 645     | -4                   |
| 128 Б  | 5000   | 3066     | 3035    | -31                  |
| 128 Б  | 10000  | 4166     | **5762**| **+1596** (+38%)     |
| 1 КБ   | 1000   | 650      | 647     | -3                   |
| 1 КБ   | 5000   | 3087     | 3017    | -70                  |
| 1 КБ   | 10000  | 4403     | **5737**| **+1334** (+30%)     |
| 10 КБ  | 1000   | 648      | 648     | 0                    |

**Выводы по пропускной способности:**
- При низкой нагрузке (1000 msg/с) оба брокера идентичны (~650 msg/с).
- При нагрузке 5000 msg/с на малых размерах результаты близки (3000–3100), но на 100 КБ Redis заметно быстрее (3066 против 2352).
- При максимальной нагрузке 10000 msg/с Redis опережает RabbitMQ на **30–86%** во всех размерах, достигая **~5700 msg/с** для сообщений ≤10 КБ. RabbitMQ максимально выдаёт ~4400 msg/с.

## 2. Потери сообщений (lost)

| Размер | Rate   | RabbitMQ (sent/received/lost) | Redis (sent/received/lost) |
|--------|--------|-------------------------------|----------------------------|
| 128 Б  | 10000  | 62595 / 62595 / 0             | 86533 / 86533 / 0          |
| 1 КБ   | 10000  | 66140 / 66140 / 0             | 86361 / 86361 / 0          |
| 10 КБ  | 10000  | 64339 / 64339 / 0             | 86378 / 86378 / 0          |
| 100 КБ | 5000   | 38955 / 35481 / **3474**      | 46115 / 46115 / 0          |
| 100 КБ | 10000  | 38223 / 34487 / **3736**      | 80888 / 63816 / **17072**  |

**Анализ потерь:**
- При размерах ≤10 КБ и нагрузке до 10000 msg/с **оба брокера работают без потерь**.
- При 100 КБ RabbitMQ начинает терять сообщения уже при 5000 msg/с (≈9% потерь), а при 10000 msg/с потери составляют ~10%.
- Redis при 100 КБ и 5000 msg/с потерь нет, но при 10000 msg/с теряет **21%** сообщений (17072 из 80888). При этом Redis смог **отправить в 2 раза больше сообщений** (80888 против 38223 у RabbitMQ), но значительная часть была потеряна.

## 3. Сравнение эффективности (отправлено vs получено)

| Размер | Rate   | Брокер     | Отправлено | Получено | % доставки |
|--------|--------|------------|------------|----------|-------------|
| 100 КБ | 10000  | RabbitMQ   | 38223      | 34487    | 90.2%       |
| 100 КБ | 10000  | Redis      | 80888      | 63816    | 78.9%       |

Redis пытался отправить гораздо больше сообщений, но consumer не успел их обработать (очередь переполнилась или истекло время). RabbitMQ из-за более низкой скорости отправки потерял меньше в процентах, но абсолютная пропускная способность Redis всё равно выше (4254 msg/с против 2287).

- На больших сообщениях (100 КБ) Redis сохраняет преимущество: 4254 против 2287 у RabbitMQ (при 10000 msg/с).

## 5. Общие выводы

1. **После выравнивания условий (многопоточный producer для обоих брокеров) Redis демонстрирует значительно более высокую пропускную способность** – до +86% при высокой нагрузке и больших сообщениях.

2. **Надёжность доставки**:
   - При разумной нагрузке (≤5000 msg/с для 100 КБ или ≤10000 msg/с для мелких сообщений) оба брокера работают без потерь.
   - На предельных режимах (10000 msg/с + 100 КБ) оба теряют сообщения: Redis теряет больше в процентах, но его абсолютная пропускная способность всё равно выше.

3. **Масштабируемость**:
   - Redis лучше использует многопоточность producer: при росте rate с 5000 до 10000 его throughput увеличивается на 88% (с 3066 до 5762), тогда как RabbitMQ – только на 36% (с 3066 до 4166).
   - RabbitMQ упирается в потолок ~4400 msg/с, Redis – ~5700 msg/с (для малых сообщений).

4. **Влияние размера сообщения**:
   - RabbitMQ чувствителен к размеру: при переходе с 10 КБ на 100 КБ его throughput падает на 45% (4283 → 2352).
   - Redis менее чувствителен: падение с 5744 до 4254 составляет 26% (при 10000 msg/с).

5. **Итоговая рекомендация**:
   - Для сценариев с высокой интенсивностью (>5000 msg/с) и сообщениями до 10 КБ **Redis предпочтительнее** (выше пропускная способность, нет потерь).
   - Для очень крупных сообщений (100 КБ) и умеренной нагрузки (≤5000 msg/с) Redis также лучше (без потерь и на 30% быстрее).
   - Если допустимы потери и нужна максимальная скорость отправки при 100 КБ, Redis всё равно даёт более высокий абсолютный throughput (4254 против 2287), несмотря на потери.
   - RabbitMQ может быть выбран, если критична гарантированная доставка при перегрузках (у него меньший процент потерь в предельных режимах), но это достигается за счёт искусственного ограничения собственной скорости отправки.

**Заключение**: В равных условиях (многопоточность) Redis обгоняет RabbitMQ по пропускной способности в большинстве сценариев, особенно при высокой нагрузке. RabbitMQ остаётся конкурентоспособным только при низкой нагрузке или в задачах, где важнее процент доставки, чем абсолютная скорость.

# Анализ результатов бенчмарка: RabbitMQ vs Redis

## Общая информация
- **Длительность каждого теста**: ~15 секунд (consumer работал 15 с, producer — 10 с согласно `main.py`, но в данных `duration` около 15 с, что соответствует времени работы consumer).
- **Целевые интенсивности**: 1000, 5000, 10000 msg/s (rate).
- **Размеры сообщений**: 128 Б, 1024 Б (~1 КБ), 10240 Б (~10 КБ), 102400 Б (~100 КБ).
- **Метрики**: `sent` – отправлено producer, `received` – получено consumer, `lost` – потери, `consumer_throughput` – реальная пропускная способность (msg/с).

---

## 1. Сравнение пропускной способности

### 1.1. При малых размерах сообщений (128 Б – 10 КБ)

| Брокер   | Rate (msg/с) | Throughput (msg/с) при 128 Б | Throughput (msg/с) при 1 КБ | Throughput (msg/с) при 10 КБ |
|----------|--------------|-----------------------------|----------------------------|-----------------------------|
| RabbitMQ | 1000         | 656                         | 659                        | 657                         |
| RabbitMQ | 5000         | 3200                        | 3201                       | 3196                        |
| RabbitMQ | 10000        | 4357                        | 4333                       | 3941                        |
| Redis    | 1000         | 605                         | 603                        | 607                         |
| Redis    | 5000         | 2614                        | 2617                       | 2640                        |
| Redis    | 10000        | 4483                        | 4489                       | 4967                        |

**Выводы:**
- На низкой нагрузке (1000 msg/с) оба брокера выдают ~600–660 msg/с, что ниже целевого значения из-за ограничений однопоточного consumer и накладных расходов Python.
- При целевой нагрузке 5000 msg/с RabbitMQ показывает **3200 msg/с**, Redis – **2600 msg/с**. RabbitMQ впереди на ~20%.
- При 10000 msg/с Redis догоняет и при размере 10 КБ даже обгоняет RabbitMQ (4967 против 3941 msg/с). При 128 Б и 1 КБ показатели близки (~4400–4500 msg/с).
- Redis лучше масштабируется при росте размера сообщения до 10 КБ (throughput растёт с 4483 до 4967 msg/с), тогда как RabbitMQ, наоборот, падает с 4357 до 3941 msg/с.

### 1.2. При больших сообщениях (100 КБ)

| Брокер   | Rate (msg/с) | Throughput (msg/с) | Потери | Примечание                              |
|----------|--------------|--------------------|--------|-----------------------------------------|
| RabbitMQ | 1000         | 657                | 0      | Стабильно                               |
| RabbitMQ | 5000         | 2309               | 1759   | Producer не успевает, часть потеряна    |
| RabbitMQ | 10000        | 2165               | 2142   | Потери растут, throughput падает        |
| Redis    | 1000         | 613                | 0      |                                         |
| Redis    | 5000         | 2586               | 0      |                                         |
| Redis    | 10000        | 4212               | 0      | Потерь нет, throughput в 2 раза выше RabbitMQ |

**Выводы:**
- На сообщениях 100 КБ RabbitMQ начинает терять сообщения при rate ≥ 5000 msg/с. Производительность падает до ~2200 msg/с.
- Redis сохраняет 100% доставку и выдаёт 4212 msg/с, что почти вдвое быстрее RabbitMQ.
- Размер сообщения сильно влияет на RabbitMQ из-за механизмов подтверждений и персистентности (даже после оптимизации). Redis легче справляется с крупными сообщениями, используя память и простой список.

---

## 2. Влияние целевой интенсивности (rate)

- **RabbitMQ**: при rate 1000 msg/с достигает ~660 msg/с (66% от цели). При rate 5000 msg/с – 3200 msg/с (64%). При rate 10000 msg/с – 4300 msg/с (43%) для малых сообщений, а для 100 КБ – только 22% (2165 от 10000). Начиная с 5000 msg/с и 100 КБ возникают потери.
- **Redis**: при rate 1000 msg/с – ~600 msg/с (60%), при 5000 – ~2600 (52%), при 10000 – ~4500–5000 (45–50%). Потерь нет во всех режимах. Redis более предсказуем.

**Точка деградации**:
- RabbitMQ деградирует при 10000 msg/с + 100 КБ (потери, падение throughput).
- Redis сохраняет работоспособность без потерь даже при 10000 msg/с и 100 КБ, хотя throughput не достигает цели из-за ограничений клиента.

## 3. Потери сообщений

| Брокер   | Размер | Rate  | Lost | % потерь |
|----------|--------|-------|------|----------|
| RabbitMQ | 100 КБ | 5000  | 1759 | 4.8%     |
| RabbitMQ | 100 КБ | 10000 | 2142 | 6.2%     |
| Redis    | любое  | любой | 0    | 0%       |

У RabbitMQ потери возникают только при максимальном размере и высокой нагрузке. Это говорит о том, что producer не успевает отправлять из-за блокировок или сетевых задержек, либо очередь на стороне брокера переполняется. Redis справляется без потерь.

---

## 4. Общие выводы

1. **Пропускная способность**:
   - При малых сообщениях (до 10 КБ) и нагрузке до 5000 msg/с RabbitMQ немного быстрее Redis (3200 против 2600).
   - При нагрузке 10000 msg/с показатели сравниваются (около 4400–4500), а на 10 КБ Redis вырывается вперёд (4967 против 3941).
   - На больших сообщениях (100 КБ) Redis уверенно лидирует (4212 против 2165 msg/с) и не теряет сообщения.

2. **Надёжность**:
   - Redis показал нулевые потери во всех тестах.
   - RabbitMQ теряет сообщения при высокой нагрузке на крупных сообщениях (до 6% потерь).

3. **Влияние размера сообщения**:
   - RabbitMQ чувствителен к размеру: при 100 КБ производительность падает в ~1.5–2 раза по сравнению с 128 Б.
   - Redis сохраняет высокую производительность даже на 100 КБ (4212 msg/с против 4483 на 128 Б – падение всего на 6%).

4. **Рекомендации**:
   - Для небольших сообщений (<10 КБ) и средних нагрузок (<5000 msg/с) оба брокера подходят, RabbitMQ даёт немного большую пропускную способность.
   - Для крупных сообщений (≥100 КБ) или высоких нагрузок (≥10000 msg/с) Redis предпочтительнее из-за отсутствия потерь и более высокой пропускной способности.
   - Если важна гарантированная доставка при пиковых нагрузках, RabbitMQ требует дополнительной настройки (увеличение prefetch, подтверждения партиями, больше потоков consumer).
